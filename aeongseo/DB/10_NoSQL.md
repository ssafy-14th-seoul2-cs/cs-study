## NoSQL(Not Only SQL)

**1. 정의**

- 기존 관계형 데이터베이스(RDBMS)의 한계 극복 위해 등장한 **비관계형** 데이터베이스 관리 시스템
- SQL만이 답은 아니다

**2. 특징**

- Schema-less(스키마 없음) : 테이블 구조를 미리 정의할 필요 없이 자유롭게 필드 추가 가능
- Scalability(수평 확장성) : 서버를 여러 대로 나누어 분산 저장 가능 (샤딩, 파티셔닝)
- High Performance(고성능) : 특정 목적에 최적화된 구조로 빠른 읽기/쓰기 가능
- Flexible Data Model(유연한 데이터 모델) : 문서, 키-값, 그래프, 칼럼 등 다양한 형태 지원
- CAP 이론 기반 : Cosistency(일관성), Availability(가용성), Partition Tolerance(분할내성) 중 2가지만 완벽히 보장 가능

**3. CAP 이론**

- Cosistency(일관성)
    - 모든 노드가 같은 시간에 같은 데이터
    - 다중 클라이언트에서 같은 시간에 조회하는 데이터는 항상 동일한 데이터
- Availability(가용성)
    - 한 노드가 다운되어도 다른 노드에 문제 생기지 않아야 함
    - 다중 클라이언트에서 모든 클라이언트의 읽기 쓰기 요청에 대하여 항상 응답 가능해야 함을 보증
- Partition tolerance(분할 내성)
    - 일부 메시지 손실해도 시스템 정상 작동
    - 지역적으로 분할된 네트워크에서 동작하는 시스템에서 두 지역간 네트워크 단절 또는 데이터 유실이 일어나도 각 지역 내 시스템은 정상 동작
- CA : 시스템 다운에도 메시지 손실 방지, 트랜잭션 사용
- DP : 데이터보다는 성능이 중요한 퍼포먼스형에 적합
- AP : 비동기화된 서비스 스토어에 적합(일관성X 서비스)

**4. 종류**

- Redis
- Apache Cassandra(아파치 카산드라)
- Hadoop(하둡)
- MongoDB(몽고DB)
- Firebase의 실시간 데이터베이스와 Cloud Firestore
- HBase

**5. 데이터 모델 유형**

| 유형 | 구조 | 대표 DB | 특징 |
| --- | --- | --- | --- |
| **Key-Value Store** | 키-값 쌍 | Redis, DynamoDB | 가장 단순하고 빠름. 캐시나 세션 저장에 적합 |
| **Document Store** | JSON/BSON 형태 문서 | MongoDB, CouchDB | 반정형 데이터 저장, 구조 유연 |
| **Column Family Store** | 행 대신 컬럼 단위 저장 | Cassandra, HBase | 빅데이터 분석에 적합 |
| **Graph Store** | 노드-엣지 구조 | Neo4j, ArangoDB | 관계 중심(소셜 네트워크, 추천 시스템)에 적합 |

**6. 장단점**

- 장점
    - 스키마 X → 유연 (비형식적인 데이터 저장 가능, 언제든지 데이터 조정, 필드 추가 가능)
    - 데이터 읽는 속도 빠름
    - ORM 프레임워크 필요 X
- 단점
    - 데이터 중복 존재 → 계속 업데이트해야 함
    - 데이터 분산 저장 → 중복저장되어 수정 시 모든 영역의 데이터 수정 필요
