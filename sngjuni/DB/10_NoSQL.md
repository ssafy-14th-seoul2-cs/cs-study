# NoSQL

## NoSQL (Not Only SQL)

---

**정의**

- **전통적인 관계형 데이터베이스(RDBMS)** 와는 다른 방식으로 데이터를 저장·관리하는 데이터베이스 시스템
- **비정형 데이터**, **대용량 데이터 처리**, **분산 시스템**에 적합하며, SQL 기반의 테이블 형식 대신 다양한 형태의 데이터를 저장

---

**핵심 내용**

1. **NoSQL의 필요성**
    - 관계형 DB는 **정형화된 스키마**와 **정규화된 데이터**가 필요하지만,
        
        **비정형/반정형 데이터**, **확장성**, **고속 처리**가 요구되는 시스템에서는 적합하지 않음
        
    - **웹 서비스**나 **빅데이터** 환경에서는 빠른 확장성과 성능이 중요한 요소로 NoSQL을 사용
2. **NoSQL의 주요 종류**
    
    
    | 유형 | 설명 | 예시 |
    | --- | --- | --- |
    | **문서 지향형(Document-Based)** | JSON, BSON 형식의 문서로 데이터를 저장 | MongoDB, CouchDB |
    | **키-값 저장소(Key-Value Store)** | 데이터를 키와 값의 쌍으로 저장 | Redis, DynamoDB |
    | **열 지향형(Column Family)** | 데이터를 열 기반으로 저장 | HBase, Cassandra |
    | **그래프 데이터베이스(Graph DB)** | 노드와 엣지 형태로 관계를 저장 | Neo4j, ArangoDB |
3. **특징**
    - **스키마가 없음**: 데이터의 형식에 제약이 적고, 동적으로 데이터를 추가할 수 있음
    - **수평 확장성**: 데이터베이스의 용량이 커질수록 **서버를 추가**하는 방식으로 확장 가능
    - **분산형 구조**: 데이터를 여러 서버에 분산 저장하여 처리 성능 및 가용성을 높임
    - **ACID 보장하지 않음**: 일부 NoSQL 시스템은 **CAP 정리(CAP theorem)** 에 따라 일관성보다 가용성과 파티션 허용을 우선시함
4. **CAP 정리 (CAP Theorem)**
    - **Consistency**(일관성): 모든 트랜잭션이 완료된 후, 데이터베이스의 모든 복제본이 동일한 값을 가짐
    - **Availability**(가용성): 언제든지 요청을 처리할 수 있어야 함
    - **Partition Tolerance**(파티션 허용): 네트워크 장애가 발생해도 시스템이 계속 운영 가능
    
    → **NoSQL**은 CAP 정리에서 **Consistency와 Availability**를 동시에 보장하기 어려움
    

---

**특징**

- **스케일 아웃(Scale-out)**: 수평 확장이 용이하여, 서버를 추가하여 용량 확장 가능
- **유연성**: 스키마리스(Schema-less)로 데이터를 추가하고 변경할 때 유연성 제공
- **높은 성능**: 대량의 데이터를 빠르게 읽고 쓰는 데 유리

---

**장단점**

- **장점**
    - **확장성**: 수평적 확장이 용이하여 대규모 데이터 처리에 유리
    - **비정형 데이터 처리**: JSON, XML과 같은 형식의 데이터를 손쉽게 다룰 수 있음
    - **빠른 쓰기/읽기 성능**: 특정 NoSQL 시스템은 **고속 읽기/쓰기**를 지원
- **단점**
    - **ACID 보장 부족**: 관계형 DB의 **트랜잭션 관리**와 같은 강력한 일관성 보장이 부족
    - **복잡성**: 여러 유형의 NoSQL이 존재하며, 각 시스템은 다른 설계 및 쿼리 모델을 요구
    - **제한된 쿼리 기능**: 복잡한 조인, 집계 등의 기능을 제공하지 않음

---

## 관련 면접 질문

- NoSQL이란 무엇인가요?
    - SQL 기반의 관계형 DB와는 다른 방식으로 데이터를 저장하는 시스템으로, 대규모 데이터 처리 및 비정형 데이터를 위한 데이터베이스입니다.
- NoSQL 데이터베이스의 장점은 무엇인가요?
    - 수평 확장성이 뛰어나고, 비정형 데이터 처리 및 빠른 읽기/쓰기 성능을 제공하며, 스키마가 유연합니다.
- NoSQL의 단점은 무엇인가요?
    - ACID 트랜잭션 보장이 부족하고, 관계형 DB에 비해 복잡한 쿼리나 조인 기능이 제한적입니다.
- CAP 정리(CAP theorem)란 무엇인가요?
    - Consistency, Availability, Partition Tolerance 중에서 두 가지를 동시에 보장할 수 있다는 이론으로, NoSQL은 이 중 Consistency와 Availability를 선택해야 합니다.
