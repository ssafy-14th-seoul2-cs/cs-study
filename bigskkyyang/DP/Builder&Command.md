# 빌더(Builder), 커맨드(Command) 패턴 정리

---

## 1. 빌더(Builder) 패턴

### 개념
복잡한 객체를 **단계별로 나누어 생성**하는 패턴.  
필수 값과 선택 옵션이 섞여 있어 생성자가 지나치게 복잡해지는 문제를 해결한다.  
“객체 생성 과정”을 분리함으로써 가독성과 유연성을 높인다.

### 왜 필요한가?
- 생성자에 넣어야 할 값이 많아지면  
  → 매개변수가 너무 많아지고 가독성이 떨어짐  
- 객체를 만드는 과정이 복잡하고 단계가 많아지면  
  → 생성 로직을 만들 때 실수가 발생하기 쉬움  
- 동일한 생성 절차를 따라가되  
  → 서로 다른 형태의 객체를 만들고 싶을 때  
- 생성 과정과 표현 방식을 분리해야 할 때

### 특징
- 객체 생성 과정을 **“조립 단계”로 분리**한다.  
- 마지막에 `.build()` 같은 메서드로 완성된 객체를 얻는다.  
- 필수 값만 먼저 받고, 선택 값을 점진적으로 추가할 수 있다.  
- 객체 생성 방식이 명확해지고 유지보수성이 좋아진다.  
- 빌더는 내부적으로 **일관된 순서로 객체를 생성**해 준다.

### 비유
- 햄버거 주문 시 빵 → 패티 → 소스 → 토핑 순서로 조립하는 과정  
- 레고 조립 설명서처럼 순차적으로 완성해 가는 느낌  
→ 복잡하지만 **정해진 절차대로 만들어야 하는 것**에 적합

## 2. 커맨드(Command) 패턴

### 개념
요청(행동)을 **객체 형태로 캡슐화**하는 패턴.  
“무엇을 실행할지”와 “누가 실행을 요청하는지”를 분리하여, 실행·취소·저장 같은 제어가 가능한 구조를 만든다.

### 왜 필요한가?
- 실행할 동작을 나중에 호출해야 할 때  
  (예: 예약 실행, 나중에 다시 불러오기)  
- 사용자가 수행한 명령을 **취소(Undo)하거나 재실행(Redo)** 해야 할 때  
- 버튼이나 메뉴에 어떤 기능을 연결할지 유연하게 교체할 필요가 있을 때  
- 동작을 로그로 남기거나 히스토리를 관리해야 할 때  
- 호출자와 실제 작업 수행자를 분리하여  
  → 기능 변경 없이 구조를 깔끔하게 만들고 싶을 때

### 특징
- 요청을 함수 호출이 아닌 “객체”로 만들어 전달한다.  
- 각 명령 객체는 `execute()`와 `undo()` 같은 고정된 인터페이스를 가진다.  
- 호출자(invoker)는 명령이 **무엇을 하는지 알 필요가 없다.**  
- 명령을 큐/스택/리스트에 저장하여  
  → 재실행, 취소, 순차 실행, 예약 실행 등이 쉬워진다.  
- UI 버튼, 단축키, 매크로 기능 등에서 자주 사용된다.

### 비유
- TV 리모컨: 버튼마다 “명령 객체”가 할당된 형태  
- 포토샵의 Undo/Redo 기능: 방금 실행한 명령을 스택에 쌓아두는 구조  
- 은행 업무에서 “입금 명령”, “출금 명령”이 각각 하나의 객체로 다뤄지는 모습  
→ “행동 자체가 하나의 독립된 객체가 된다”는 점이 핵심
